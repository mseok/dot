# Section Mapping

Map the report section to its real function before drafting.

## 1. 사업 추진 상세내용

Typical function:

- explain what problem was addressed
- explain what was designed or developed
- explain how the developed item is intended to be used
- explain how it is or will be checked
- explain why it matters for later expansion

Recommended paragraph order:

1. problem definition
2. model/system overview
3. development details
4. utilization path
5. validation path
6. domestic/global expansion path

Do not force all six into every subsection. Use the subset required by the
writing guide.

## 2. 추진실적 및 성과

Typical function:

- show what was achieved against the baseline plan
- interpret the result level
- identify stronger and weaker areas

Recommended order:

1. metric frame
2. measured results
3. interpretation
4. implications and next improvement target

## 3. 결과활용 및 확산계획

Typical function:

- show where the result can be used
- show who can use it
- show what infrastructure or service path exists
- show domestic and global expansion logic

Recommended order:

1. immediate use cases
2. service or deployment path
3. public/industry adoption path
4. domestic/global expansion logic

## 4. 성과지표/평가표 대응

If a section is likely to be read against an evaluation rubric:

- surface implementation, evidence, and utility clearly
- make reviewer inference easy
- avoid copying rubric language verbatim unless needed
- do not mention `진척도` or `평가점수` directly unless the form requires it

## 5. Comparison Writing

When comparing to AF3 or another baseline:

- keep the comparison axis explicit
- distinguish `implementation difference`, `input condition difference`,
  `evaluation condition difference`
- do not let rhetoric outrun the benchmark conditions
